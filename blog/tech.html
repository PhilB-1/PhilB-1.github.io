Test Driven Development

February 21th, 2016

When you are writing tons of lines of code you may want to have a way of making sure it is doing what you want it to do. Now we could run the code with different parameters, but that may not work with every scenario and is not the best way of displaying. When you want clear and coherent descriptions of whether you code is working properly you'll want to write some codes. This is called test driven development. You write test to ensure that you code is working problem and you develop and refactor your code based on these test. These test usually also print our a state if they fail, so that it will easily describe to anyone what was supposed to happen, but didn't. In our lessons we've seen two examples of test, one for Ruby called rspec, and one for JavaScript called assert. If you look below you will see an example of each.

require_relative 'my_solution' describe Die do describe '#initialize' do it 'expects a single argument' do expect(Die.instance_method(:initialize).arity).to eq 1 end it 'raises an ArgumentError if list of labels is empty' do expect { Die.new([]) }.to raise_error(ArgumentError) end end

This first example is an example of a Ruby rspec file. The first require relative is referring to the solution file you are testing. Describe refers to the class or method you are testing. It is followed by a small string of what you are expecting to happen. The expect line then runs a piece of code to test that the thing it expects happen actually happens.

function assert(test, message, test_number) { if (!test) { console.log(test_number + "false"); throw "ERROR: " + message; } console.log(test_number + "true"); return true; } assert( (adam instanceof Object), "The value of adam should be an Object.", "1. " )

Now we will discuss our example of JavaScript. The first part in any JavaScript test is to create your assert function. This function accepts a few arguments and allows you run all your test by calling the assert function. It describes that if the test doesn't work, print error, test number, false, and the message of what failed to the console. Otherwise it will print the test number and true. The next section is the actual test. It has the arguments needed for the above function and the actual test in the parenthesis. In this example it is checking to see if adam is an object. These test are great for when your code runs, but is not doing what it is intended to do.
